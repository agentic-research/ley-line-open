# `cloister-spec/` — capability spec directory layout

**Status:** Draft (2026-05-18, paired with bead `cloister-94893e`, L6 of
the substrate-IDL track).
**Audience:** anyone authoring a new capability spec under
`cloister-spec/`, or a second implementor reading one and wondering what
shape they're meant to find. Operators wiring a capability into a real
cloister deployment want `QUICKSTART.md` (per-capability); spec authors
want this doc.

**Non-goals:** This is a *layout doc*. It codifies the on-disk shape of
a conformant capability-spec directory and the CI contract that gates
drift. It is **not** the migration plan that moves
`interlace-spec/0.1.0/` under `cloister-spec/` — that is bead `NL5`
in the substrate-IDL track, intentionally deferred.

## What "today" vs "going forward" mean in this doc

The repository has two capability specs as of 2026-05-18, and they
have incongruent shapes. This doc:

- **Describes** what each looks like today (so you can read the
  existing files without surprise).
- **Prescribes** the shape every *new* capability spec MUST take
  going forward, including the eventual fill-in of the older one.
- **Recommends** a single canonical home — see §Migration plan — and
  flags that the file moves themselves are out of scope for this doc.

When this doc says **today**, it describes current on-disk reality.
When it says **going forward** or uses RFC-2119 **MUST / SHOULD / MAY**,
it is prescribing the contract new capabilities are held to.

## §Required layout

**Going forward**, every capability spec under `cloister-spec/` MUST
follow this shape:

```
cloister-spec/<cap>/<v>/
├── README.md            # the spec proper — invariants, non-goals, wire summary
├── QUICKSTART.md        # operator walkthrough — wire it into a real cluster in 5 min
├── wire/                # per-surface canonical-bytes docs (one .md per surface)
│   ├── <surface-1>.md
│   ├── <surface-2>.md
│   └── ...
├── test-vectors/        # canonical inputs + expected digests (JSON-as-carrier)
│   ├── README.md        # explains the JSON-as-carrier convention
│   ├── <surface-1>.json
│   └── ...
├── ref-impl-<lang>/     # at least one second implementation (typically Python)
│   ├── pyproject.toml   # or Cargo.toml / go.mod
│   ├── <pkg>/           # the implementation
│   └── conformance/     # `run.py` (or equivalent) — exits non-zero on drift
└── VECTORS.sha256       # pinned digest of every test-vectors/*.json file
```

Concretely, here's `cloister-spec/credential-isolation/v1/` going
forward (the parts marked `✓` exist today; the parts marked `□` are
the gap L7 fills):

```
cloister-spec/credential-isolation/v1/
├── README.md                          ✓
├── QUICKSTART.md                      ✓
├── wire/
│   ├── proxy-envelope.md              ✓
│   ├── injection-strategies.md        □  (referenced in README)
│   ├── receipt-commitment.md          □  (referenced in README)
│   └── error-codes.md                 □  (referenced in README)
├── test-vectors/                      □  (L7 populates)
│   ├── README.md                      □
│   ├── proxy-envelope.json            □
│   ├── injection-strategies.json      □
│   ├── receipt-commitment.json        □
│   └── allowedSubs-glob.json          □
├── ref-impl-py/                       □  (L7 populates)
│   ├── pyproject.toml                 □
│   ├── credisol/                      □
│   └── conformance/run.py             □
└── VECTORS.sha256                     □  (regenerated by L7)
```

**Required files**

| File | RFC-2119 | Why |
|---|---|---|
| `README.md` | MUST | The spec proper. Status, audience, non-goals, wire summary, versioning, errata. |
| `QUICKSTART.md` | SHOULD | Operator wiring guide. Five-minute path from zero to a working capability call. Required for capabilities with operators (i.e. all of them today). |
| `wire/*.md` | MUST (≥1 file) | Per-surface canonical-bytes specification. One `.md` per signed/canonical surface. |
| `test-vectors/*.json` | MUST | The contract. Canonical inputs + expected digests. JSON-as-carrier, NOT JSON-as-spec — see `interlace-spec/0.1.0/README.md` §"Note on test-vector format". |
| `ref-impl-<lang>/` | MUST (≥1) | A second implementation. If only one implementation exists, the wire isn't a wire. Python is the default reference language. |
| `VECTORS.sha256` | MUST | Pinned SHA-256 of every `test-vectors/*.json` file. Output of `shasum -a 256 test-vectors/*.json`. |

**Naming rules**

- `<cap>`: kebab-case, English, no underscores. E.g. `credential-isolation`, `substrate-manifest`. Not `credIsol` or `cred_isolation`.
- `<v>`: semver-shaped. Two patterns are accepted today:
  - `v1`, `v2`, ... — major-only. Used by `credential-isolation/`. Appropriate when minor/patch revisions never break wire bytes (errata only).
  - `0.1.0`, `0.2.0`, ... — full semver. Used by `interlace-spec/`. Appropriate when minor versions add backwards-compatible surfaces (new optional record types, new non-critical OIDs) and patch versions ship clarifying errata.
- **Going forward**, prefer full semver (`0.1.0`). The `v1`-only shape is grandfathered; it works, but full semver makes the channel pattern below cleaner.

## §Versioning

This section echoes the channel-and-lockfile pattern documented in
`prior-art/nixpkgs.md` §Decision, adapted to capability specs.

**Channel cadence.** Capability versions follow a *coordinated release*
discipline, not a continuous-delivery one. A capability `<v>` directory
is **frozen** once it has shipped test-vectors and a passing ref-impl;
post-freeze changes are either:

- **Errata-only** — README/wire-doc wording fixes that do NOT change
  any byte of any test vector. Allowed in-place. README MUST gain an
  errata entry (see `interlace-spec/0.1.0/README.md` §Errata for the
  shape). The frozen test vectors are the load-bearing truth; prose
  bends to match.
- **Backwards-compatible additions** (minor bump, e.g. `0.1.0 → 0.1.1`)
   — new optional fields, new non-critical extension OIDs, new
  disclosure record types. Consumers MUST tolerate unknown values.
  Bump to a new `<v>` directory.
- **Wire-breaking** (major bump, e.g. `0.1.0 → 0.2.0`) — anything
  that changes a canonical-bytes input. New `<v>` directory; in-flight
  draft work lives at `<cap>/<vnext>-draft/` (see
  `interlace-spec/0.2.0-draft/` for the existing pattern).

**Content-addressed pinning.** Per `prior-art/nixpkgs.md` Adopt list,
the `VECTORS.sha256` file is the per-capability equivalent of Nix's
`narHash`. A consumer that pins a capability by `(<cap>, <v>,
VECTORS.sha256)` has byte-exact knowledge of which spec it depends on.
The future `art.lock` manifest (substrate-IDL NL-tier work) will roll
these per-capability digests up to a substrate-channel digest
(`art-2026.05.0` etc.).

**When to bump major vs minor**

- **Major** if any of these change:
  - Canonical-bytes encoding at any signed surface (CBOR shape, DER
    field set, UTF-8 concat order).
  - Extension OID renumbering.
  - Grammar of any string field consumers parse (scope, paths,
    cursor format).
  - The set of MUST-NOT-commit fields in a receipt.
- **Minor** if changes are additive and unknown-tolerant consumers
  keep working:
  - New optional record types (JSONL disclosure).
  - New non-critical extension OIDs in the same OID arc.
  - New optional bundle fields beyond existing integer keys.
- **Patch** (errata-only) if no byte of any test vector changes.

The Nixpkgs analogy: a major bump is a new release channel
(`nixos-25.11`); a minor bump is a backport into the current channel;
a patch is a security update that doesn't touch the build closure.

## §Drift contract

**Today.** `cloister/.github/workflows/interlace-spec-drift.yml`
(filed under bead `cloister-af1290`) gates every PR that touches
`interlace-spec/**` against three rules:

1. **Vector-byte drift.** Every `test-vectors/*.json` file's SHA-256
   matches its entry in `VECTORS.sha256`. Drift fails the build with
   a message telling the dev to either revert or bump the spec
   version. This is the byte-level contract.
2. **Ref-impl conformance.** The Python reference under
   `interlace-spec/0.1.0/ref-impl-py/` is run on every PR via
   `uv run python conformance/run.py`. A divergence between the ref
   impl and the pinned vectors is treated as a spec-level finding,
   not a CI auto-fix.
3. **Version-string sanity.** Any change to a file under
   `test-vectors/` that is not paired with a touch of the same
   version's `README.md` in the same PR fails fast. Vector changes
   are spec-breaking by definition, so the version string at the
   top of the README MUST move.

The contract is intentionally noisy: silent vector drift would let
cloister and the spec diverge invisibly, breaking the byte-compat
guarantee the spec exists to enforce.

**Going forward, two extensions land:**

- **Schema-shape drift.** Bead `cloister-94fdba` (L3, "schema-bridge
  diff subcommand") adds `schema-bridge diff <old> <new>` and wires
  it into a CI step alongside the existing vector-byte gate. Where
  today's gate catches "the bytes of the test vectors changed," the
  L3 gate catches "the *shape* of the schema changed" — added /
  removed / renamed / retyped fields tiered FILE / PACKAGE / WIRE
  per Buf's rule taxonomy (`prior-art/buf.md` §Axis 3). These are
  complementary: a field rename with no vector change today slips
  past CI; under L3, it lands in the `Renamed` tier and forces a
  reviewer to acknowledge the rename intentionally.
- **Per-capability gating.** Today only `interlace-spec/**` is gated.
  Going forward, every capability under `cloister-spec/<cap>/<v>/`
  with a populated `test-vectors/` MUST have an equivalent
  drift-check workflow (or reuse a single parameterized one). L7
  (bead `cloister-1a3b...`, conformance vectors for credential-
  isolation/v1) is expected to add the second instance.

**What the drift contract does NOT enforce** (today or going forward):

- Spec correctness — only spec stability. A test vector that pins
  the wrong bytes is still byte-stable; reviewers, not CI, catch
  semantic errors.
- Cross-capability consistency — that
  `credential-isolation/v1`'s lease envelope actually inherits from
  `interlace-spec/0.1.0`'s, not a redefinition. Cross-capability
  references are prose today; making them mechanically checkable is
  follow-on work.

## §Migration plan

**The asymmetry.** Today the repo has two top-level spec directories:

```
cloister/
├── cloister-spec/
│   └── credential-isolation/v1/     ← under cloister-spec/
└── interlace-spec/
    ├── 0.1.0/                        ← top-level, NOT under cloister-spec/
    └── 0.2.0-draft/
```

This is real architectural inconsistency. Three options are on the
table:

- **(a)** Move `interlace-spec/` under `cloister-spec/interlace/` so
  the whole spec tree has one root. Proposed unblocker for non-leaf
  NL5.
- **(b)** Keep `interlace-spec/` at top level because it is the
  *foundational substrate* every other capability consumes —
  treating it as a peer-of-cloister-spec, not a child. Other
  capabilities live under `cloister-spec/`.
- **(c)** Move `cloister-spec/*` up to top level, parallel to
  `interlace-spec/`. So `credential-isolation/v1/` at the repo root.

**Recommendation: (a) — move `interlace-spec/` under
`cloister-spec/interlace/`.**

Reasoning:

1. **The CONSUMES arrow runs from credential-isolation → interlace,
   not the other way.** `credential-isolation/v1/README.md` already
   says it "consumes" `interlace-spec/0.1.0/`. The fact that
   interlace is foundational does NOT make it not a capability spec
   — it makes it the *first* capability spec. Foundationality is a
   property of the dependency graph, not of the file layout.
2. **One root simplifies the drift workflow.** Today
   `interlace-spec-drift.yml` is paths-scoped to `interlace-spec/**`
   and L7's hypothetical `credisol-drift.yml` would be scoped to
   `cloister-spec/credential-isolation/**`. Under (a), a single
   parameterized workflow can iterate every
   `cloister-spec/<cap>/<v>/` and apply the same three rules.
3. **The Nixpkgs analogy points here.** In `prior-art/nixpkgs.md`,
   Nixpkgs is *the* package tree; there's no parallel "foundational
   packages live elsewhere." A new substrate-channel manifest
   (`art-2026.05.0`) wants a single root to enumerate.
4. **(b) preserves the asymmetry as policy.** That's a worse outcome
   than (a) because every new capability author has to memorize the
   exception. (c) is the inverse of (a) and has equivalent
   bookkeeping cost without the "foundational at the root" framing
   working in its favor — the dependency graph still flows the same
   way, so (c) gains nothing structural.

**Out of scope for this doc.** This doc is *layout*, not migration.
The move itself (file relocations, updating
`interlace-spec-drift.yml` paths, updating `ref-impl-py` imports,
updating cloister's `src/routes/lease-middleware.ts` comment
references) is bead NL5 in the substrate-IDL track. NL5's
acceptance criterion can now cite this doc's recommendation rather
than re-deciding.
